# Plan — infra-loop-timing

## Goal
Measure wall-clock time per YOLO loop stage to find non-AI bottlenecks, using timing data we *already emit* rather than adding new instrumentation.

## Scope
**In scope** (exactly the two declared `deliverable_paths`):
- `experiments/infra-loop-timing/timing_extract.py` — read-only timing extractor.
- `experiments/infra-loop-timing/REPORT.md` — generated findings report.

**Explicitly NOT in scope:**
- No changes to `tick_tock.yml`, `council.py`, or any pipeline file (no new instrumentation — the council JSON timestamps and GitHub Actions step boundaries already exist).
- No `infra-loop-timing/` directory at repo root (infra ticks live only under `experiments/`).
- No network writes; the gh path is read-only (`gh run list/view`).

## Approach
The experiment hypothesis (from `nb-2026-04-16-fix-bottleneck-not-ai-speed`) is: *AI speed may not be the loop bottleneck.* To test it we need a per-stage wall-clock decomposition. Two timing sources already exist with **zero added instrumentation**:

1. **Local council JSON timestamps (primary, always available).** Every gate writes `<project>/council_<gate>.json` with an ISO `timestamp`. The four gates (plan→implementation→tests→outcome) of a single build give inter-gate intervals: build/fix time + council latency. This source needs no network and is mineable offline right now (~18 projects on disk).

2. **GitHub Actions step durations (CI-only, optional).** `gh run view <id> --json jobs` returns each workflow step's `name` + `startedAt`/`completedAt`. This captures the *non-AI* stages the council JSON cannot see — checkout, pip install, npm install, git push — which is exactly where non-AI bottlenecks hide.

**Data-source decision (answers PLAN council_focus "GitHub Actions vs instrumenting council.py"):** We use BOTH existing sources and instrument NOTHING. Instrumenting `council.py` would only measure council latency and miss the surrounding non-AI stages (install/checkout/push) that the hypothesis targets; it would also add code to the hottest path. GHA step boundaries + council JSON timestamps together cover the full loop for free.

**Robustness to YAML reformat (answers IMPLEMENTATION council_focus):** the gh path reads `--json jobs` (structured step *names* + timestamps), never parses workflow YAML line-by-line, so a YAML reformat cannot break parsing. The local path keys off filenames + JSON fields, also YAML-independent.

**Subtasks & sequencing:**
1. `parse_iso` + file collection helpers (no deps).
2. `extract_local_timings` — group gate files per project, compute deltas, flag same-session coherence. (depends on 1)
3. `aggregate_local` — medians, per-interval stats, objection totals. (depends on 2)
4. gh helpers: `gh_available`, `list_recent_runs`, `extract_gh_timings`, `aggregate_gh` — all degrade gracefully to empty when no token/binary. (depends on 1)
5. `identify_bottlenecks` — combine aggregates + scan `tick_tock.yml` for missing dependency caching. (depends on 3,4)
6. `render_report` + `main`/argparse + `--selftest`. (depends on all)

**Same-session coherence (critical data-quality guard):** gate JSON files are overwritten across builds, so a project dir can hold gate files from *different dates* (observed: 61-day inter-gate deltas). `extract_local_timings` computes intervals ONLY when all present gates fall within `--max-session-gap-min` (default 360); otherwise the project is marked `coherent: false` and excluded from interval aggregates (but still listed).

## File Layout
- `experiments/infra-loop-timing/timing_extract.py` — NEW, ~230 LOC.
- `experiments/infra-loop-timing/REPORT.md` — NEW, generated by `--report`, then lightly annotated.

## Function Map
`experiments/infra-loop-timing/timing_extract.py`:
- `parse_iso(ts) -> datetime|None` — tolerant ISO parse.
- `collect_council_files(roots) -> dict[str, dict[str,str]]` — project → {gate: path}.
- `extract_local_timings(files, max_gap_min) -> list[dict]` — per-project gate timestamps, inter-gate deltas (sec), objection counts, `coherent` flag.
- `aggregate_local(records) -> dict` — median/total per gate-interval over coherent builds; objection totals.
- `gh_available() -> bool` — gh binary + token present.
- `run_cmd(args) -> (int, str)` — subprocess wrapper, never raises.
- `list_recent_runs(workflow, n) -> list[str]` — run IDs via `gh run list --json`.
- `extract_gh_timings(run_ids) -> list[dict]` — per-run step name→duration via `gh run view --json jobs`.
- `aggregate_gh(runs) -> dict` — median duration per step name.
- `detect_uncached_installs(yaml_path) -> list[str]` — flag install steps with no `actions/cache`/`cache:` sibling.
- `identify_bottlenecks(local_agg, gh_agg, yaml_findings) -> list[dict]` — ranked findings.
- `render_report(log) -> str` — markdown.
- `selftest() -> int` — runs local extraction, asserts invariants, returns exit code.
- `main() -> int` — argparse: `--source {local,gh,both}`, `--runs N`, `--out`, `--report`, `--max-session-gap-min`, `--selftest`.

## Security
Read-only tool. Threat surface: (1) reads JSON/YAML on disk — wrapped in try/except, never executes content. (2) `gh` subprocess — fixed argv lists (no `shell=True`, no string interpolation of untrusted data into a shell), only `gh run list/view`. (3) no secrets read or written; relies on ambient `GH_TOKEN` already in the runner. (4) writes only to the two declared paths. No CSP/trust-boundary concerns (no HTML/network server).

## UI
CLI only. `--report` prints a one-line summary + writes REPORT.md. Default run writes `timing_log.json`. `--selftest` prints PASS/FAIL lines.

## Guide
`--help` documents every flag. REPORT.md opens with a plain-language methodology + the data-source decision so a reader understands what each number means (and what it conflates: inter-gate interval = build + council latency, not council alone).

## Edge Cases
- No gh token / no gh binary → gh section empty, `sources_used` notes it, exit 0 (local still works).
- Project with <2 gate files → listed, no intervals.
- Gate files from different dates → `coherent: false`, excluded from aggregates.
- Malformed/empty JSON → skipped with a counted `skipped_files`.
- `gh` returns non-JSON / error → caught, run skipped.
- Empty council corpus → aggregates are `null`, report says "no coherent builds found".
- Clock-skew / negative deltas → flagged `coherent: false`.

## Test Strategy
- Pre-filter: `python3 -c "import ast; ast.parse(open(p).read())"` on the script.
- `python3 timing_extract.py --selftest` — asserts: ≥1 coherent build found locally, all deltas ≥0 for coherent builds, JSON round-trips, report renders non-empty. (>5 local projects available — satisfies the "≥5 real datapoints" intent; the gh-mode 5-real-runs test is deferred to CI where `GH_TOKEN` exists, and is documented as such in REPORT.md.)
- `python3 timing_extract.py --source local --report` — produces `timing_log.json` + `REPORT.md` with real numbers.
- OUTCOME: REPORT.md names ≥1 concrete non-AI bottleneck (expected: uncached pip/npm install on every hourly run).
