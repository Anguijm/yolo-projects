# eval-managed-agents

Benchmark **Claude Managed Agents** (first-party server-run orchestration) against
this repo's **manual** tick-tock orchestration, to decide whether managed agents
can simplify tick-tock dispatch logic.

Adopted from Phase 4 experiment `nh-2026-04-09-claude-managed-agents`.

## The premise (verified against the live API)

Managed Agents is a real first-party surface (`managed-agents-2026-04-01`):

- `POST /v1/agents` — create a **persisted, versioned** Agent (model, system
  prompt, tools). Create once, reference by id.
- `POST /v1/sessions` — start a **stateful** Session referencing that agent in an
  environment. Anthropic **runs the agent loop** and provisions a **per-session
  container** where the agent's tools (bash/read/write/edit/…) execute.
- The session **streams events**; you send user messages / tool results in.

That contrasts directly with how the YOLO builder works today: the tick-tock
driver owns the whole loop, resends full conversation history every turn (the
Messages API is stateless), hosts tool execution itself, and hand-rolls
retry / `pause_turn` / context compaction.

## What this measures

A dimension-by-dimension comparison of **what code the repo owns** under each
orchestration model, plus an **orchestration-ownership delta** — the loop /
context / tool-hosting / retry surface the repo could *stop* maintaining if it
adopted managed agents, weighed against the explicit 4-gate council discipline it
would have to re-express.

## Run it

Static comparison (deterministic, offline, no key — this is what regenerates
`results.md`):

```bash
python3 benchmark.py            # human summary + results.md
python3 benchmark.py --json     # machine-readable JSON
```

Live wall-clock + token benchmark (needs a real key, **never run from cron**):

```bash
ANTHROPIC_API_KEY=... python3 benchmark.py --live
```

The live path uses the documented SDK surface (`agents.create` once →
`environments.create` → `sessions.create` → `events.stream`/`.send`, with the
correct terminal idle/terminated break gate) and accumulates token usage from
`span.model_request_end` events. It is **gated** and dormant in CI because a
managed session runs for minutes and bills real tokens — incompatible with the
autonomous builder's synchronous one-shot tick.

## Verdict (see `results.md`)

**ITERATE / DEFER.** Managed agents removes the agent-loop, context-passing,
tool-hosting, and error-recovery code that dominates manual orchestration — but
the tick-tock pipeline's value is its explicit, synchronous, git-audited 4-gate
discipline, which is exactly where a server-run, minutes-long managed session
fits worse. Recommended next step: prototype the **council-gate sub-step alone**
as a managed session with `always_ask` tool gates, measured live with a real key,
before changing any dispatch code.
