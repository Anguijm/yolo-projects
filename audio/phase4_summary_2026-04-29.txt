Hey, here's a plain-language rundown of the Phase 4 experiments that have come in since we last talked on April 25th.

Quick context first. Phase 4 is the part of the system where a cron job watches a handful of YouTube channels every day, pulls down new videos, and turns each one into a tiny experiment proposal. The idea is, instead of you having to watch hours of AI content, the system harvests the ideas, files them in a backlog, and you decide later which ones are worth actually building. Today's count: one hundred twenty nine total experiments, twenty nine still sitting in the backlog, and the cron last ran successfully a few hours ago.

Since our last chat, eleven new experiments have landed. One was auto-skipped because it was just news commentary about Apple, no real action. So that leaves ten real proposals. I'll walk you through them grouped by theme.

Theme one: running multiple AI coders side by side. There are three of these. Mark Kashef had two ideas. The first is to run Claude and OpenAI's Codex in parallel on the same codebase, each one chewing on different tasks, so you get more done in the same wall-clock time. The second is more refined: use Claude as the planner, the one who breaks the work down, and let Codex be the executor that actually writes the code. Two specialists instead of one generalist. Then a fellow named NateBJones suggested wiring OpenAI's Responses API in as a background dev task runner, basically a free always-on intern that handles the boring stuff like ticket triage and doc updates while you work on the real thing.

Theme two: moving the whole loop to the cloud. Two of these came from MLOps Community. The first proposes running every YOLO loop iteration inside an ephemeral cloud sandbox, something like Modal, E2B, or Daytona, so each agent gets a clean isolated environment instead of running on your laptop. The second is the natural follow-on: if you're going to run agents in the cloud, you need real observability, structured trace logs on every tool call, every reasoning step, every token spent. Otherwise you're flying blind.

Theme three: smarter use of Claude Code itself. Two experiments here, both from NateHerk. The first is to adopt a structured CLAUDE.md file plus a curated library of slash commands, so every session starts on rails instead of you re-explaining the project from scratch. The second is to run Claude Code in headless mode, meaning no interactive UI at all, just a CLI you can pipe inputs into and chain together inside shell scripts.

Theme four: self-improving agents. Two related ideas from a creator named David Ondrej. The first is a self-modifying agent that reads its own performance logs and rewrites its own prompts and tools between runs, learning from its own mistakes. The second is the scaffolding underneath that: a lightweight evaluation harness that scores each agent run and feeds those scores back into the next run as context. The harness is the boring foundation; the self-evolving agent is the flashy thing on top.

Theme five: model routing. One experiment, again from NateBJones. Instead of using a single model for everything, benchmark GPT five point five, Claude, and Gemini on different task classes, reasoning, code, long-context retrieval, and route each task to the model that's empirically strongest for that class.

Now, the honest part. All ten of these are still sitting in the backlog. None of them have been promoted to the tick queue, none have been built, none have been adopted or discarded yet. The current tick queue, the stuff you've already approved for actual building, is from earlier batches: a bare-bones fifty-line agent loop, a model swap benchmark, a strategic niche audit, a managed agents evaluation, and session checkpointing.

If I had to point at the most likely next adoptions from this fresh batch, I'd say the structured CLAUDE.md plus slash commands one is the lowest effort and highest immediate payoff, and the eval harness is the natural prerequisite for anything self-improving. The cloud sandbox idea is high effort and probably should wait. The parallel Claude plus Codex experiments are interesting but overlap with work you've already got queued.
