# `.harness/` — guides + steering loop

This directory holds the **guides** (feedforward controls — persona files, security checklist, plan, halt docs) and the **steering loop artifacts** (learnings, session state, council reports). The matching **sensors** live elsewhere: gitleaks, lint, typecheck, tests in CI, and the council runner that reads these guides.

Generated by `harness init`; specialize per-repo by editing the persona scope sections and the security checklist.

## Layout

```
.harness/
├── README.md                 # this file
├── halt_instructions.md      # circuit-breaker docs (guide)
├── model-upgrade-audit.md    # checklist when swapping models (guide)
├── active_plan.md            # current work plan + Repository Impact Map (guide)
├── learnings.md              # steering-loop artifact: append-only KEEP/IMPROVE/INSIGHT/COUNCIL log
├── session_state.json        # auto-updated by post-commit hook (steering-loop artifact)
├── yolo_log.jsonl            # append-only audit trail (steering-loop artifact)
├── last_council.md           # most recent council report (steering-loop artifact)
├── council/                  # persona files = the council's guides
└── scripts/
    ├── council.py            # the inferential sensor's runner
    ├── install_hooks.sh      # one-time setup (sets core.hooksPath)
    ├── requirements.txt      # python deps for council
    └── security_checklist.md # injected into Security reviewer prompt (guide)
└── hooks/
    └── post-commit           # updates session_state + yolo_log
```

## One-time setup

```bash
bash .harness/scripts/install_hooks.sh                # wire post-commit hook
pip install -r .harness/scripts/requirements.txt      # python deps for council
bash scripts/setup-secrets.sh                         # sync GEMINI_API_KEY etc. to repo
```

## Repository Impact Map (pre-plan grounding)

Before writing a plan, run `harness map "<feature description>"` to scan the actual codebase and produce a grounded list of files, symbols, and tests likely affected. Without this step, plans drift toward hallucinated paths and invented APIs — the industry-named "vague in, vague out" anti-pattern.

The output is written to `.harness/active_plan.md` as a `## Repository Impact` block at the top. The human writes the rest of the plan beneath it.

## Local council usage

```bash
# review a plan file (must be tracked in git)
python3 .harness/scripts/council.py --plan .harness/active_plan.md

# review the working diff vs origin/main
python3 .harness/scripts/council.py --diff --base origin/main
```

The CI workflow (`.github/workflows/council.yml`) does the same thing automatically on every PR.

## Halt

To stop all council automation: `echo "reason" > .harness_halt` at repo root. See `halt_instructions.md`.

## Steering loop

When a sensor catches something the guides should have prevented (recurring BLOCK on the same surface, recurring CI failure pattern, a class of bugs the council keeps missing) — **the guide is wrong, not the sensor.**

Update the guide:
- Persona scope is too generic → edit the persona's `## Scope`.
- Doctrine is missing → add a section to `CLAUDE.md`.
- Anti-pattern needs banning → add to `security_checklist.md` or a persona's non-negotiables.

Append the lesson to `learnings.md` so it persists across sessions. Never edit prior entries; append-only.

## Specializing for the repo

The personas in `council/` ship as generic skeletons. **They will produce generic critiques until specialized.** Edit each persona's `## Scope` section to describe:
- This repo's actual modules and their boundaries.
- The contracts other code or repos depend on.
- The security surface (auth model, trust boundaries, secret handling).
- The cost drivers (LLM calls, external services, storage).

Same for `scripts/security_checklist.md` — generic items at the top, repo-specific items added below them.
